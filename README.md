# یدکی — فروشگاه اینترنتی قطعات یدکی خودرو

مستند وضعیت پروژه بر اساس سند نیازمندی اولیه (بندهای ۱ تا ۸). این فایل خلاصه‌ی دقیقی از چیزهای پیاده‌سازی‌شده و چیزهای باقی‌مانده است.

## پشته‌ی فنی

- **بک‌اند:** Laravel + Laravel Passport (احراز هویت API)
- **فرانت‌اند:** Next.js 14 (App Router) + React 18 + TypeScript
- **UI:** MUI v7 + Emotion (RTL با `stylis-plugin-rtl`)
- **مدیریت state:** Zustand (سبد خرید، احراز هویت)
- **پرداخت:** زرین‌پال
- **پیامک:** کاوه‌نگار (Verify Lookup API)
- **کاروسل:** Embla Carousel
- **تقویم:** react-multi-date-picker (جلالی)

### بند ۴ — فرآیند اختصاصی سفارش (کامل)

جریان کامل طبق سند:

1. ثبت اولیه‌ی سفارش بدون پرداخت (`pending_review`)
2. اعلان به ادمین/انبار برای بررسی موجودی
3. حالت اول: تأیید کامل (همه‌ی اقلام موجود) → پیامک + لینک پرداخت زماندار (۳۰ دقیقه)
4. حالت دوم: ویرایش جزئی توسط ادمین (حذف/کاهش اقلام ناموجود) → needs_customer_confirmation → پیامک به مشتری
5. تأیید مشتری روی سبد اصلاح‌شده → لینک پرداخت زماندار
6. امکان تلاش مجدد برای پرداخت بعد از انقضای لینک (بدون نیاز به سفارش جدید)
7. فیلتر وضعیت سفارش‌ها (رهاشده/پرداخت‌نشده/تأییدنشده) در پنل مدیریت + جدول با صفحه‌بندی سمت سرور

## نکات فنی مهم برای ادامه‌ی توسعه

- **RTL:** پروژه از stylis-plugin-rtl استفاده می‌کند که حتی property های فیزیکی (left/right/transform: translateX) را خودکار برعکس می‌کند. برای موقعیت‌دهی دقیق پیکسلی همیشه از property های منطقی (insetInlineStart/insetInlineEnd) یا style خام React (نه sx) استفاده کنید.
- **الگوی SSR:** صفحات اصلی فروشگاه (هومپیج، لیست محصولات، جزئیات محصول، مقالات) به‌صورت Server Component با async fetch هستند؛ فقط تعامل‌های کاربر (فیلتر، سبد خرید، فرم‌ها) کامپوننت کلاینتی جداگانه دارند.
- **نام‌گذاری الگوهای پیامک کاوه‌نگار:** باید فقط از آندرلاین (\_) استفاده شود، نه خط‌تیره (-) — وگرنه خطای ۴۳۱ ("ساختار کد صحیح نمی‌باشد") رخ می‌دهد. مقادیر توکن هم نباید فاصله داشته باشند.
- **صفحه‌بندی جدول‌ها:** الگوی استاندارد شامل TablePagination واقعی MUI + فیلتر سمت سرور است (نه فیلتر کلاینتی روی یک لیست کامل).
